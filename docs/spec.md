# 求職論壇彙整擴充 — 設計規格（Design Spec）

日期：2026-07-13
狀態：v1 設計定案，待開發

## 1. 目標

使用者在 104／1111 看某家公司職缺時，不用自己一個個跳去 PTT、Dcard 翻，擴充自動把該公司在論壇上的相關文章（標題＋可點連結）就地列在職缺頁旁邊。

**一句話定位**：把「查一家公司要開好幾個分頁翻論壇」壓縮成「打開職缺就看到」。

## 2. 使用者與情境

- 主要使用者：正在求職、會在 104/1111 篩公司、習慣去 PTT/Dcard 查風評的人。
- 情境：使用者在 104 打開某公司或某職缺頁 → 擴充在頁面上出現一塊面板 → 列出 PTT（Tech_Job、Salary 等）與 Dcard 上提到這家公司的文章標題 → 點標題直接新分頁跳到原文。

## 3. 範圍

**v1 要做（In scope）**
- 在 104／1111 職缺頁與公司頁偵測出「公司名」
- 用公司名去 PTT、Dcard 搜尋
- 面板列出結果：文章標題 + 來源標記（PTT/Dcard）+ 版名 + 可點超連結
- 點擊在新分頁開原文

**明確不做（Out of scope，YAGNI）**
- AI 摘要（暫緩）
- 後端伺服器 / 資料庫（純前端，路線 A）
- 自建職缺資料庫、帳號系統、收藏、跨裝置同步
- 內容轉載（只顯示標題＋連結，導流回原站，不搬全文）

**下一階段（v2 再說，先設計好不擋路）**
- 偏好篩選：上班環境 / 同事氛圍 / 薪水升遷等面向（用關鍵字組合搜尋實現）
- 更多來源（Glassdoor、比薪水、面試趣連結、Google 評論）
- 更多求職站（518、CakeResume、Yourator 等）

## 4. 架構總覽（路線 A：純前端擴充，無後端）

Manifest V3 Chrome 擴充。所有動作都在使用者瀏覽器內完成，沒有伺服器。

```
┌─────────────────── 使用者的瀏覽器 ───────────────────┐
│                                                      │
│  104/1111 職缺頁                                      │
│   └─[content script] 偵測公司名 ──┐                  │
│                                    ▼                  │
│                          [background service worker] │
│                            ├─ 公司名正規化            │
│                            ├─ PTT 抓取器 ─→ ptt.cc    │
│                            └─ Dcard 抓取器 ─→ dcard.tw │
│                                    │                  │
│              合併結果 ◀────────────┘                  │
│                 │                                     │
│   104 頁面 ◀─[content script 注入面板] 顯示結果       │
│                                                      │
└──────────────────────────────────────────────────────┘
```

為什麼抓取放在 background service worker 而不是 content script：MV3 擴充在 background 用 `host_permissions` 發跨網域請求可繞過 CORS（content script 直接 fetch dcard.tw 會被 CORS 擋）。Dcard 的同源需求另見 §8。

## 5. 模組拆解

每個模組單一職責、介面清楚、可獨立測試。

| 模組 | 職責 | 輸入 → 輸出 |
|---|---|---|
| **公司偵測器**（content script）| 從 104/1111 DOM 抓出公司名 | 頁面 DOM → 公司名字串 |
| **公司名正規化器** | 把「台灣積體電路製造股份有限公司」轉成可搜尋的名字 | 原始公司名 → 搜尋詞陣列（如 `["台積電","台灣積體電路"]`）|
| **PTT 抓取器** | 打 PTT 搜尋、解析文章列表 | 搜尋詞 → `[{標題, 版, 網址}]` |
| **Dcard 抓取器** | 打 Dcard 搜尋、解析 | 搜尋詞 → `[{標題, 看板, 網址}]` |
| **結果合併器** | 合併去重、排序（依相關度/時間）| 多來源結果 → 統一列表 |
| **面板 UI**（content script）| 把結果注入頁面顯示 | 統一列表 → 畫面上的面板 |

抓取器都遵守同一個介面 `search(term) → Result[]`，之後加 Glassdoor、比薪水只要多寫一個抓取器，其他不用動。

## 6. 資料流（一次查詢的生命週期）

1. 使用者打開 `104.com.tw/company/xxxx` 或 `/job/xxxx`
2. 公司偵測器讀 DOM → 得到「台灣積體電路製造股份有限公司」
3. 正規化器 → `["台積電", "台灣積體電路"]`
4. background 平行呼叫 PTT 抓取器、Dcard 抓取器
5. **漸進顯示**：PTT 快，先回先顯示；Dcard 好了再補進面板（不讓使用者乾等）
6. 合併器去重、排序
7. 面板 UI 顯示；點標題 → 新分頁跳原文

## 7. 核心難點：公司名比對策略

這是準確度的生死，單獨設計。

**問題**：104 上是全名「台灣積體電路製造股份有限公司」，論壇上是「台積電 / tsmc / 護國神山」。

**v1 策略（由簡到繁，先做前兩層）**
1. **去法律後綴**：砍掉「股份有限公司/有限公司/企業社」等 → 「台灣積體電路製造」
2. **同時搜全名與短名**：全名和去後綴版都丟去搜，聯集結果
3. **別名字典（小規模人工維護）**：維護一份熱門公司對照表（全名 → 常用簡稱），命中就加簡稱去搜。隨使用累積擴充，涵蓋 Top 幾百家就能蓋掉大部分查詢。

**準確度取捨**：v1 只顯示「標題含搜尋詞」的結果，接受一點雜訊，由使用者看標題自己判斷——因為我們只列標題＋連結、不做結論，誤報成本低。精準過濾留到 v2。

## 8. 各來源抓取細節

### PTT（實測完全可行）
- 端點：`https://www.ptt.cc/bbs/{板}/search?q={詞}`，板先鎖 `Tech_Job`、`Salary`、`Soft_Job`
- 回傳 HTML，解析文章列表的標題與 `/bbs/{板}/M.{id}.html` 連結
- **實測（2026-07-13）**：伺服器端 curl（無 cookie、無特殊 header）即回 200，Tech_Job / Salary 各抓到 20 筆/頁，標題連結解析無礙 → 擴充端零風險
- 少數板（Gossiping）需 `over18=1` cookie，v1 先不納入

### Dcard（結論：不內嵌抓取，改一鍵跳站內搜尋）

**Stage 2 spike 實測結論（2026-07-14，已定案）**：放棄從擴充抓 Dcard，改為面板底部提供
「在 Dcard 搜尋『{公司}』」連結，開新分頁到 `https://www.dcard.tw/search?query={詞}`。
不抓取、不需要 dcard.tw 權限、不會被 Cloudflare 擋、無 ToS 風險。

證據（同一台瀏覽器、同一時段）：

| 情境 | 結果 |
|---|---|
| 伺服器端 curl 打 API | 403（`Cf-Mitigated: challenge`）|
| 瀏覽器**導航**到 API 網址 | 200 JSON（真瀏覽器自動解 CF 挑戰）|
| 從 dcard.tw 頁面**同源 fetch** API | 200、有資料 ✅ |
| **擴充 background fetch**（冷，無 clearance）| 403 挑戰頁 |
| **擴充 background fetch**（溫，同源 fetch 剛剛才 200）| **仍然 403** ❌ |

關鍵：**不是 cookie 有沒有帶的問題**。擴充背景請求的來源是 `chrome-extension://`，
Cloudflare 依 Origin/Referer/Sec-Fetch 等特徵判定為跨站而重新挑戰。要繞過就得偽造標頭
＝刻意規避反機器人機制，ToS 風險高且隨時會壞，不採用。

次要理由：Dcard 搜尋是**模糊比對**，實測搜「全景軟體」首筆回傳「觀光&日文 大三上選課分享」，
精準度遠不如 PTT 的標題包含比對，內嵌價值本來就打折。

### Dcard（伺服器端被 Cloudflare 擋；瀏覽器端待真環境驗）
- 端點：`https://www.dcard.tw/service/api/v2/search/posts?query={詞}&limit=30`，回傳 JSON
- 文章網址格式：`https://www.dcard.tw/f/{看板}/p/{文章ID}`
- **實測（2026-07-13）**：伺服器端 curl 回 **403**，header 為 `Cf-Mitigated: challenge`、`Server: cloudflare`，body 是 Cloudflare managed challenge（Turnstile）頁。加瀏覽器 header 無效——擋的不是 header，是「沒過 JS 挑戰、沒有有效 `cf_clearance` cookie、指紋不像真瀏覽器」。
- **判讀**：這是伺服器端悲觀情境，**不能推論到瀏覽器端**。擴充跑在使用者真實瀏覽器裡，帶著逛過 Dcard 的 `cf_clearance` cookie＋真瀏覽器指紋，Cloudflare 通常直接放行。
- **瀏覽器端策略**：讓請求對 `dcard.tw` **同源**（fetch 走在 Dcard context，或 background 帶對 cookie/origin），使用者逛過 Dcard 就有 clearance。
- **兩個必須真環境驗的前提**：
  1. **同源請求**：從 104 頁跨域直打 Dcard API 可能被 CORS/challenge 卡，需真環境確認。
  2. **冷啟動**：使用者從沒開過 Dcard、無 `cf_clearance` 時第一次呼叫仍可能碰 challenge。
- **備案**：Dcard 若在真環境不穩 → 改用 `site:dcard.tw {公司名}` 搜尋取得文章連結，或退成「一鍵跳 Dcard 搜尋」不內嵌。

</details>

### 權限（manifest）
- `host_permissions`: `*://*.ptt.cc/*`（**只有 PTT**；Dcard 改用連結跳轉，不需要權限）
- `content_scripts` match: `*://www.104.com.tw/job/*`, `*://www.104.com.tw/company/*`（1111 待 Stage 3）

## 9. UI／面板位置

104 職缺頁大致版位（0–100 為頁面寬度）：

```
0                                              100
┌────────────────────────────┬──────────────────┐
│                            │                  │
│    職缺主內容（職務說明）      │   ← 我們的面板    │
│    公司名、薪資、條件         │   浮在右側        │
│                            │  ┌──────────────┐│
│                            │  │論壇風評(13) – ││ ← 標題列
│                            │  │[Dcard][Google]││ ← 外站連結列
│                            │  │▸ 新聞 (26)    ││ ← 新聞區，預設收合
│                            │  │─PTT/Tech_Job─ ││
│                            │  │• [心得] 面試.. ││
│                            │  │─PTT/Salary─── ││
│                            │  │• [討論] 分紅.. ││
│                            │  └──────────────┘│
└────────────────────────────┴──────────────────┘
```

- 形式：右側浮動卡片，可收合。標題列顯示「論壇風評 (筆數)」＋公司名
- **外站連結列（標題列下方，兩顆並排）**：`Dcard 搜尋`（query=公司名）、`Google 搜尋`（query=公司名 評價）。
  用 `primarySearchTerm()` 取最貼近人話的詞（有綽號用綽號，否則中文全名去後綴；不用剝過行業詞的版本）。
  hover 顯示實際送出的搜尋字串。
- **新聞區（連結列下方，預設收合）**：所有 `[新聞]` / `Re: [新聞]` 開頭的文章**不分板**抽到這一區，
  點標題列展開/收合。理由：上市公司的新聞（股價、訴訟、產業動向）量大且與求職關聯低——
  實例：大立光 44 筆中 26 筆是新聞，會淹掉真正有用的 [心得]/[請益]。判斷邏輯在 `src/lib/categorize.js`，
  由 handler 標記 `isNews`，面板只負責分組顯示。
- 內容：**扣掉新聞後**的結果依板分組，每筆一行＝標題（省略號截斷），整行可點開新分頁
- 載入中：顯示「搜尋中…」

### Stage 1 實作結果與學到的事（實測後補記）

在真實 104 頁實測後，發現「公司名比對」比預期更關鍵，Stage 1 因此比原計畫多做了幾層正規化：

- **偵測公司名**：職缺頁用 JSON-LD `hiringOrganization.name`（實測最穩）；公司頁用 `h1`。原計畫的 `a[href*="/company/"]` 備援實測會抓到導覽列「推薦好公司」，已移除。
- **104 是 Vue SPA**：站內點職缺是前端換頁、不重載，content script 只在硬載入跑一次 → 面板時有時無。已改為監看 pushState/replaceState/popstate + 1s 輪詢兜底，每次換頁重跑；JSON-LD 常事後注入，偵測不到會重試 ~5s。
- **公司名 ≠ PTT 叫法**（正規化 `src/lib/normalize.js`，4 層）：
  1. 拆底線：104 用「英文品牌_中文全名」（KEYENCE_台灣基恩斯…），拆開分別搜。
  2. 去法律後綴（可夾中間，如 …股份有限公司楊梅廠 → 中華汽車工業）。
  3. 剝行業類別詞（克勞德科技 → 克勞德），防護：縮完 ≥2 字且非通用詞（台灣/中華…黑名單）。
  4. 綽號對照表 `src/lib/aliases.js`：縮寫綽號無法用規則推得（台灣積體電路製造→台積電），人工維護 ~39 家大公司，加之前先驗證 PTT 有料。
- **搜尋板**（`src/handler.js` `PTT_BOARDS`）：`Tech_Job`、`Soft_Job`、`Salary`、`job`。Tech/Soft 偏理工，Salary 跨領域，job 是通用求職板（含文組）。
  - 實測結論：外商板 `Foreign_Inv` 低活躍、外商公司多為 0 筆（KEYENCE 在此 0 筆、真正 35 筆在 Tech_Job/Salary），不加。產業別板（Bank_Service/Marketing/Law/media-chaos/Nurse…）留給未來「依 104 產業別自動選板」，不要全域全搜（浪費請求、可能被 PTT 限流）。
- **debug**：content script 會在頁面 console 印 `[求職論壇風評] 偵測公司 / 搜尋詞 / 結果數`。

## 12. 風險與對策

| 風險 | 影響 | 對策 |
|---|---|---|
| Dcard Cloudflare 擋 | Dcard 抓不到 | 已確認伺服器端被 managed challenge 擋（`Cf-Mitigated: challenge`）；瀏覽器端因帶真實 cookie＋指紋很可能可過，但同源與冷啟動需真環境實測；最壞退路＝`site:dcard.tw` 搜尋或一鍵跳轉 |
| 104/1111 改版，DOM 抓不到公司名 | 偵測失效 | 公司偵測器獨立模組、選擇器集中管理，改版只改一處 |
| 論壇版面/API 變動 | 抓取器壞掉 | 抓取器隔離、各自可獨立修；有 log |
| 公司名比對誤報/漏抓 | 結果不準 | v1 接受雜訊、只列標題導流；別名字典漸進優化 |
| 法律 / ToS | Dcard ToS 可能禁止程式化存取 | v1 只顯示標題＋連結、導流回原站（類搜尋引擎行為），不轉載全文；用使用者自身瀏覽器存取；上架前再確認各站條款 |
| 被當爬蟲限流 | 抓取被擋 | 路線 A 天生分散（每人自己 IP、量小）；面板加簡單快取（同公司短時間內不重抓）|

## 12.5 決策記錄：哪些資料放遠端、哪些留程式碼（2026-07-16）

已放遠端的只有**綽號對照表**（`data/aliases.json`，見 §14）。曾討論是否把「行業詞剝除／組織前後綴／產業→板對照」也搬遠端以免送審，結論如下。

**判斷準則不是「會不會影響全部人」（下載都影響全部人），而是「一筆壞資料的爆炸半徑」＋「能不能自動驗證」。**

| 資料 | 一筆壞掉的波及 | 可驗證性 | 決策 |
|---|---|---|---|
| 綽號對照表 | 窄——只影響名字含該字串的公司，且只「多加」搜尋詞不會削短 | 可驗（比對字非空、綽號為字串陣列），壞則退回內置 | **放遠端** ✅ |
| 行業詞／組織前後綴 | 全域——套用在每家公司的字首字尾，一個常見字（如「電」）會一次削短大量公司名成過度通用詞 | 幾乎無法自動驗（「電」格式合法但語意危險）| **留程式碼、走送審** |
| 產業→板對照 | 小——壞了頂多搜錯板（多雜訊或搜到空板），不會削短公司名 | 尚可 | **留程式碼**（見下）|

**為什麼行業詞不搬**：爆炸半徑全域、驗不了、又很少改，正好該留在有測試＋稽核（`scripts/audit.mjs`）把關的程式碼裡。而且**綽號表本身就是萬用逃生口**——任何規則漏掉的公司都能用一筆綽號遠端補上（例：即使沒有「餐飲」剝除規則，也能加 `{王品餐飲→王品}`），所以「免送審修 gap」的需求綽號表已滿足，不需要冒險搬字尾規則。

**為什麼產業→板也不搬（YAGNI）**：涵蓋已大致完整（53/284，其餘產業本就無對應板；可用的 PTT 板幾乎掃完收齊）；整個專案只改過 3 次且全在初期；未來改動頻率趨近零。搬遠端要多養一套載入＋驗證＋退回機制，收益（一年可能 0～1 次、每次省一趟可合併的送審）撐不起這複雜度。真實使用若證明常需加板再回來搬。

**一句話**：高爆炸半徑（字尾規則）留程式碼走送審；低爆炸半徑且高頻（綽號）放遠端。
